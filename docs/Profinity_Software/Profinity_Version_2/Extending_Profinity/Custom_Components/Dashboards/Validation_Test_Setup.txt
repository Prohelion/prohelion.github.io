Dashboard Examples Validation Test - Setup and Usage Guide
==========================================================

OVERVIEW
--------
This document explains how to set up and run the automated validation test that checks
all YAML dashboard examples in the documentation against the ui-schema.json schema.

The test ensures that all examples in the documentation are compliant with the current
schema, helping maintain documentation quality and preventing outdated examples.

LOCATION
--------
Test file: Profinity/Profinity-Web-GUI/ClientApp/src/helpers/__tests__/validate-dashboard-examples.test.ts

Schema file: Profinity/Profinity-Engine/UserInterface/Factories/ui-schema.json

Documentation: Profinity-Docs/docs/Profinity_Software/Profinity_Version_2/Extending_Profinity/Custom_Components/Dashboards/

SETUP
-----
1. Ensure both workspaces are available:
   - Profinity workspace (contains the test and schema)
   - Profinity-Docs workspace (contains the documentation)

2. The test uses an absolute path to find the documentation:
   /Work/Profinity-Docs/docs/Profinity_Software/Profinity_Version_2/Extending_Profinity/Custom_Components/Dashboards/

   If your workspace is in a different location, you'll need to update the DOCS_PATH
   constant in the test file.

3. Install dependencies (if not already done):
   cd Profinity/Profinity-Web-GUI/ClientApp
   npm install

4. The test automatically uses the mock schema copied during test setup:
   src/mocks/ui-schema.json (copied from Profinity-Engine during test:setup)

RUNNING THE TEST
----------------
The test is excluded from the general test suite to avoid breaking regular test runs.

To run the validation test explicitly:

  cd Profinity/Profinity-Web-GUI/ClientApp
  npm test -- validate-dashboard-examples.test.ts

WHAT THE TEST DOES
-----------------
1. Scans all .md files in the Dashboards documentation directory
2. Extracts all YAML code blocks (```yaml or ```yml)
3. Wraps partial examples in a complete dashboard structure if needed
4. Validates each example against ui-schema.json using AJV
5. Reports:
   - Total number of examples found
   - Number of valid examples
   - Number of invalid examples
   - Details of validation errors for invalid examples

EXPECTED OUTPUT
---------------
The test will output:
- A count of markdown files found
- For each file, the number of YAML blocks extracted
- A validation summary showing:
  * Total examples: X
  * Valid: Y
  * Invalid: Z
- A list of invalid examples with their file locations and error details

TROUBLESHOOTING
---------------
1. Test not found:
   - Ensure you're in the ClientApp directory
   - Check that the test file exists at the path shown above

2. "Docs directory does not exist":
   - Verify the Profinity-Docs workspace is available
   - Check that the path in DOCS_PATH matches your workspace location
   - Update DOCS_PATH in the test file if needed

3. "Failed to load schema":
   - Ensure the test:setup script has run (copies schema to src/mocks/)
   - Check that ui-schema.json exists in Profinity-Engine

4. No examples found:
   - Verify markdown files exist in the documentation directory
   - Check that YAML blocks use ```yaml or ```yml syntax

5. Many invalid examples:
   - This is expected if documentation hasn't been updated to match schema changes
   - Fix examples in the documentation files to resolve validation errors
   - Common issues:
     * Old structure: lamps: items: lampgroup (should be lamps: lampgroup:)
     * Old structure: readouts: items: readout (should be readouts: readout:)
     * Wrong property: charttype (should be type)
     * Missing required properties

INTEGRATION WITH TEST SUITE
----------------------------
The test is excluded from the general test suite (npm run test) to prevent:
- Breaking regular test runs if documentation is temporarily out of sync
- Slowing down the test suite
- Requiring the Profinity-Docs workspace to be available for all tests

To include it in the general test suite, remove this line from jest.config.cjs:
  '/__tests__/validate-dashboard-examples\\.test\\.ts$'

MAINTAINING THE TEST
--------------------
When updating the documentation:
1. Run the validation test after making changes
2. Fix any validation errors in the examples
3. Ensure all new examples are schema-compliant
4. Update this guide if the test location or usage changes

The test should be run:
- Before committing documentation changes
- When the schema is updated
- Periodically to catch documentation drift

NOTES
-----
- The test wraps partial examples (e.g., just a "row:" component) in a complete
  dashboard structure for validation
- Only YAML blocks explicitly marked with ```yaml or ```yml are extracted
- Plain ``` code blocks are also extracted (assumed to be YAML in dashboard docs)
- The test uses the same validation logic as CustomDashboardFactory in the C# code
