Overview

The Profinity Dashboard system allows you to create dynamic, data-driven user interfaces using YAML configuration files. This guide will help you understand how to structure your dashboard, configure components, and implement data bindings.

Creating Custom Dashboards

To create a custom dashboard, you need to create a Profinity Custom Component that includes both:

  1. DBC File - Defines the CAN bus messages and signals that your component will use

  2. YAML File - Defines the dashboard layout and component bindings, if you don't provide a Dashboard Profinity will provide a default one for you.

The Custom Component editor provides dedicated editors for both files, which can be accessed by clicking on the editor icon on the toolbar

  • DBC Editor - For defining CAN messages and signals

  • Dashboard Editor - For creating the YAML dashboard configuration

Schema Validation

The dashboard editor uses a schema to validate your YAML Dashboard file. The editor understands this Dashboard schema and will:

  • Provide syntax highlighting and autocomplete

  • Show validation errors in real-time

  • Refuse to load dashboards that don't comply with the schema

  • Prevent invalid configurations from being saved

This ensures that your dashboard configurations are always valid and will work correctly when deployed.

Getting Started

  1. Create a new Custom Component in the Profinity system

  2. Define your CAN messages and signals in the DBC editor or load in an exiting DBC file

  3. Create your dashboard layout in the YAML editor or load in an existing Dashboard

  4. The system will validate your configuration against the schema

  5. Once valid, your dashboard will be available for use

How Profinity Dashboards Accesses Data

Profinity dashboards can call in to the main Profinity engine to get realtime values using the Profinity Data Store (PDS) format. The structure for accessing the Profinity Data Store is as follows.

Component Name Placeholders

Throughout this guide, you'll see {COMPONENT_NAME} used in data binding examples. This is a placeholder that gets replaced at runtime with the actual component name. For example:

  • {COMPONENT_NAME}.BusMeasurement.BusVoltage becomes WaveSculptor 22.BusMeasurement.BusVoltage

  • {COMPONENT_NAME}.[Property].Status becomes Prohelion D1000 Gen 1.[Property].Status

This allows you to create reusable dashboard templates that can be applied to different components without modifying the YAML configuration. The system automatically substitutes the placeholder with the specific component name when the dashboard is rendered.

DBC Messages and Signals

The primary data source uses DBC (Database CAN) format for CAN bus messages and signals:

  • {COMPONENT_NAME}.BusMeasurement.BusVoltage - Accesses a signal from a CAN message

  • {COMPONENT_NAME}.Status.Online - Accesses a status signal from a CAN message

Direct C# Properties

Use [Property] to access C# properties directly:

  • {COMPONENT_NAME}.[Property].Status - Accesses a C# property directly

  • {COMPONENT_NAME}.[Property].Configuration.Version - Accesses nested properties

  • {COMPONENT_NAME}.[Property].PackData.NodeStatusColourText[1] - Accesses an array element

  • {COMPONENT_NAME}.[Property].PackData.BatteryMilliVolts - Accesses battery voltage data

  • {COMPONENT_NAME}.[Property].State.Controller.CurrentState.Name - Accesses nested state information

Time Series Data

Use [TimeSeries] to access time-series data for charts and historical displays:

  • [TimeSeries].{COMPONENT_NAME}.BusMeasurement.BusCurrent - Time series data for charts

  • [TimeSeries].{COMPONENT_NAME}.VelocityMeasurement.VehicleVelocity - Historical velocity data

The system automatically handles the different data source types and provides appropriate data binding capabilities for each.

The Four Core Dashboard Elements

A dashboard is defined as a collection of full-width items arranged vertically. Each top level can be one of four types:

  • Titlebar: Header section with status lamps and navigation menus

  • Row: Layout containers that can hold multiple components

  • Accordion: Collapsible sections for organizing content

  • Footer: Bottom section with navigation menus

The dashboard uses a hierarchical structure where rows can contain groups, and groups can contain individual components. This allows for flexible layouts that can adapt to different screen sizes and content requirements.

Data Binding

Data binding allows you to connect your dashboard components to dynamic data sources. The binding system supports data transformation, type conversion, and value mapping.

Basic Binding Structure

All bindings are defined as arrays of binding objects:

yamlwide760

Binding Parameters

Required Parameters:

  • target (string): The property to bind to (e.g., "value", "label", "color")

  • source (string): The data source path (e.g., "data.temperature", "status.online")

Optional Parameters:

  • toType (string): Data type conversion ("number", "string", "boolean")

  • gain (number): Multiplicative scaling factor

  • offset (number): Additive offset

  • invert (boolean): Whether to invert the value

  • mapToText (object): Text mapping configuration

Best Practices

  1. Use Descriptive Source Paths: Make your data paths clear and consistent

  2. Handle Data Types: Always specify toType when the source data type might be ambiguous

  3. Test Transformations: Verify that gain/offset calculations work as expected

  4. Plan for Missing Data: Consider how your components behave when data is unavailable

  5. Use Meaningful Labels: Map values to human-readable text when possible

Troubleshooting

Common Issues

  1. Component Not Updating: Check that your data source is providing updates

  2. Incorrect Values: Verify gain/offset calculations and data types

  3. Missing Components: Ensure all required parameters are provided

  4. Layout Issues: Check direction settings and CSS classes

Debugging Tips

  1. Start with static values before adding bindings

  2. Test individual components before combining them

  3. Use simple data sources initially

  4. Verify data paths match your actual data structure

This guide provides the foundation for creating dynamic, data-driven dashboards using the Profinity system. Experiment with different components and binding configurations to create the dashboard that best meets your needs.
