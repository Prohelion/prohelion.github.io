Dashboard Examples Validation Script - Setup and Usage Guide
==============================================================

OVERVIEW
--------
This document explains how to set up and run the automated validation script that checks
all YAML dashboard examples in the documentation against the ui-schema.json schema.

The script ensures that all examples in the documentation are compliant with the current
schema, helping maintain documentation quality and preventing outdated examples.

LOCATION
--------
Script file: Profinity-Docs/scripts/validate-dashboard-examples.js

Schema file: ../Profinity/Profinity-Engine/UserInterface/Factories/ui-schema.json
             (relative to Profinity-Docs root)

Documentation: Profinity-Docs/docs/Profinity_Software/Profinity_Version_2/Extending_Profinity/Dashboards/

SETUP
-----
1. Ensure both workspaces are available:
   - Profinity workspace (contains the schema)
   - Profinity-Docs workspace (contains the script and documentation)

2. The script expects the Profinity project to be a sibling directory:
   /Work/Profinity/Profinity-Engine/UserInterface/Factories/ui-schema.json

   If your workspace is in a different location, you'll need to update the SCHEMA_PATH
   constant in the script file.

3. Install dependencies:
   cd Profinity-Docs/scripts
   npm install

RUNNING THE SCRIPT
------------------
To run the validation script:

  cd Profinity-Docs
  node scripts/validate-dashboard-examples.js

Or using npm:

  cd Profinity-Docs/scripts
  npm run validate-dashboards

WHAT THE SCRIPT DOES
---------------------
1. Scans all .md files in the Dashboards documentation directory
2. Extracts all YAML code blocks (```yaml or ```yml)
3. Wraps partial examples in a complete dashboard structure if needed
4. Validates each example against ui-schema.json using AJV
5. Reports:
   - Total number of examples found
   - Number of valid examples
   - Number of invalid examples
   - Details of validation errors for invalid examples
6. Writes detailed error information to validation-errors.json

EXPECTED OUTPUT
---------------
The script will output:
- Schema location confirmation
- A count of markdown files found
- For each file, the number of YAML blocks extracted
- A validation summary showing:
  * Total examples: X
  * Valid: Y
  * Invalid: Z
- A list of invalid examples with their file locations and error details
- Detailed error information written to validation-errors.json

TROUBLESHOOTING
---------------
1. Script not found:
   - Ensure you're in the Profinity-Docs directory
   - Check that the script file exists at scripts/validate-dashboard-examples.js

2. "Failed to load schema":
   - Verify the Profinity workspace is available as a sibling directory
   - Check that ui-schema.json exists at:
     ../Profinity/Profinity-Engine/UserInterface/Factories/ui-schema.json
   - Update SCHEMA_PATH in the script if your workspace structure is different

3. "Docs directory does not exist":
   - Verify the documentation directory exists
   - Check that the path in DOCS_PATH matches your workspace location
   - Update DOCS_PATH in the script if needed

4. No examples found:
   - Verify markdown files exist in the documentation directory
   - Check that YAML blocks use ```yaml or ```yml syntax

5. Many invalid examples:
   - This is expected if documentation hasn't been updated to match schema changes
   - Fix examples in the documentation files to resolve validation errors
   - Common issues:
     * Old structure: lamps: items: lampgroup (should be lamps: lampgroup:)
     * Old structure: readouts: items: readout (should be readouts: readout:)
     * Wrong property: charttype (should be type)
     * Missing required properties

6. Module not found errors:
   - Run `npm install` in the scripts directory
   - Ensure package.json exists and has the required dependencies (ajv, yaml)

MAINTAINING THE SCRIPT
----------------------
When updating the documentation:
1. Run the validation script after making changes
2. Fix any validation errors in the examples
3. Ensure all new examples are schema-compliant
4. Update this guide if the script location or usage changes

The script should be run:
- Before committing documentation changes
- When the schema is updated
- Periodically to catch documentation drift

NOTES
-----
- The script wraps partial examples (e.g., just a "row:" component) in a complete
  dashboard structure for validation
- Only YAML blocks explicitly marked with ```yaml or ```yml are extracted
- Plain ``` code blocks are also extracted (assumed to be YAML in dashboard docs)
- The script uses the same validation logic as CustomDashboardFactory in the C# code
- Exit code 0 = success (all examples valid), exit code 1 = failure (some examples invalid)
